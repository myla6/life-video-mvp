# 上场必备

## 1. 30 秒 / 60 秒项目介绍

### 1.1 最短版（30 秒，开场必背）

> 我做了一个宝宝满月纪念短片的 MVP，项目名 `life-video-mvp`。用户上传照片、填宝宝信息、选 BGM，一键生成竖屏 mp4。  
> 前端 Next.js；API 用 Route Handler 创建异步任务写 Postgres，文件落本地 storage；Python Worker 轮询任务表，ffmpeg 按模板 JSON 合成；前端轮询状态完成后预览下载。  
> 第一周用 DB 轮询打通 **手动 E2E**，数据库 Postgres 跑 Docker，ORM 用 Prisma。

### 1.2 端到端一句（你特别熟这句）

> 我负责满月短片 MVP 的 **端到端链路**：用户上传照片 → 创建异步任务 → Worker ffmpeg 合成 → 前端轮询 → 预览下载，**本地手动 E2E 已跑通**。

### 1.3 架构版（60 秒，追问时用）

> 这是典型的 **长任务 + 状态机** 架构：POST 只接单返回 jobId，不在 HTTP 里跑 ffmpeg。Worker 独立进程消费任务，status 从 created 到 completed。  
> 数据上 **Job 表** 存任务元信息和状态，**JobAsset 表** 存每张照片的路径和 sortOrder；真实文件在 storage，DB 只存相对路径。  
> 我前端背景，这次在同一条业务链上补了 API、Prisma、Docker 本地 Postgres，以及和 Python Worker 的协作边界。

### 1.4 和奇觅 / 公司经历衔接（有游戏广告视频经验时）

> 我在公司参与了 [奇觅](https://qimi.com/) 游戏广告 AI 视频平台的 **前端**，熟悉上传→配置→生成→等待→下载这条产品流。  
> 之后发现朋友有满月、结婚纪念等 **生活化 1 分钟短片** 需求，和公司场景不同。我独立做了 `life-video-mvp`，负责需求、前后端和视频合成链路，用 **模板编排 + ffmpeg** 生成成片，而不是自研大模型，也不是仿即梦。

### 故事 1：为什么做（痛点 + 用户）

**可讲版本：**

> 朋友宝宝满月需要发朋友圈的纪念短片，现有做法是手工剪或找模板工具，要么太重要么效果不稳定。  
> 我在奇觅做过游戏广告视频前端，知道「上传素材→选配置→等生成→下载」这套流程，但公司场景是 B 端投放，算法团队在公司内部，个人没法复刻。  
> 朋友要的是 **生活场景 1 分钟成片**：照片多、有音乐字幕、竖屏发朋友圈——本质是 **模板化智能剪辑**，用 ffmpeg + 规则就能 MVP，不需要自研大模型。  
> 所以我选了 **满月模板先做**，需求真实、范围可控。

**要点：** 生活化 vs 奇觅/即梦；真实用户；个人可落地。

### 故事 2：需求怎么砍（边界）

**可讲版本：**

> 第一版我只做 **宝宝满月** 一个模板，片长随照片数量计算，不凑固定 60 秒。  
> **砍掉的：** 自由时间轴剪辑、用户登录支付、微信小程序、AI 文生视频、结婚模板（放第二版）、横屏高清。  
> **保留的：** 5～15 张照片、可选短视频、表单信息、3 首 BGM、异步生成、预览下载。  
> 原则是开工包里的：**先纵向打通全链路，再打磨效果和第二个模板**。  
> 朋友访谈还没系统做，但竖屏 720×1280、照片为主这些定案是明确的。

**要点：** 能说出 In Scope / Out of Scope；「先打通再完美」。

---

### 故事 3：技术取舍（为什么用这个方案）

**可讲版本：**

> **Next.js 全栈** 而不是单独 NestJS/Java：个人 MVP，API 和前端同 repo，我前端背景学习成本最低。  
> **Python Worker + ffmpeg** 而不是 Node 合成：视频生态在 Python，且 Day 2 已验证 `render.py`。  
> **Postgres + Prisma** 而不是 Mongo：任务状态适合关系型；Prisma 类型友好、迁移进 git。  
> **第一周 DB 轮询** 而不是上 Redis：少一个依赖，先 E2E 跑通；第二周再演进 Redis 队列。  
> **本地 storage** 而不是先 OSS：调试快，结构不变，上线再换 OSS。  
> **没用 TanStack Start / Kafka**：MVP 不增加不确定性；Kafka 是更重的消息方案，朋友量级用 Redis 就够。

**要点：** 每个选型都有「MVP 优先跑通」的理由 + 演进路径。

### 故事 4：线上 / 开发问题（怎么定位、怎么修）

**可讲版本（选 1～2 个真实经历）：**

**经历 A — 任务一直「排队中」**

> E2E 时提交成功但永远排队。我按层排查：Network 里 POST 201 正常 → Prisma Studio 里 Job 是 created → 发现 **Worker 进程没起**。  
> 起了 `python main.py` 后 status 变为 rendering → completed。  
> 意识是：**长任务不是 API bug，可能是消费者没跑。**

**经历 B — Worker 连不上数据库**

> Worker 报 `Connection refused` 或 `invalid dsn schema`。  
> 前者是 Docker Postgres 没开；后者是 Prisma 连接串带 `?schema=public`，psycopg2 不支持，在 config 里去掉 query 参数。  
> 排查路径：**Worker 终端 → Docker → .env 对齐**。

**经历 C — 第一次跑通 E2E**

> 验证标准是手动 E2E：5 张照片 → Studio 里 1 Job + 5 JobAsset → storage/uploads 有文件 → Worker 日志出现「完成」→ 任务页能播放下载。  
> 不是只看接口 200，而是 **用户故事整段成立**。

**要点：** 分层排查（Network → 终端 → Studio → 磁盘）；有具体现象和结论。

---

### 故事 5：AI 怎么用（你负责什么、AI 负责什么）

**可讲版本：**

> 我前端背景做全栈 MVP，用 AI 加速 **boilerplate 和 ffmpeg 参数**，但我自己定 **范围、验收标准和 E2E**。  
> 我不说「做下一步」，我说「Worker 接入后，手动 E2E 必须从上传到 mp4 下载；Studio 里 status 变 completed」。  
> AI 改完我按四层验收：Network、Next 终端、Worker 终端、Prisma Studio + storage 目录。  
> 例如 Redis：我让 AI 用面试口吻解释 **为什么第一周不用、第二周怎么加**——我要的是 **决策叙事**，不是背八股文。  
> **我负责：** 产品边界、表结构设计、接口契约、跑通和验收。  
> **AI 负责：** 模板代码、Prisma route、Python Worker 骨架、文档整理。

**要点：** 证明你会用 AI 但不空转；强调验收标准。

# 分题型应答

## 4. 架构 & 技术取舍

**Q：整体架构？**

> 浏览器 → Next.js（页面 + Route Handler）→ Postgres 记任务 → storage 存文件 → Python Worker 轮询 DB → ffmpeg 合成 → 前端轮询 GET 任务 → 预览下载。

**Q：为什么 Next 当后端不用 Java？**

> MVP 个人项目，Route Handler 和前端同仓库，改 API 和页面一次提交。合成重活用 Python。上线后结构不变，只是部署到 ECS。

**Q：为什么 Python Worker 不和 API 写在一起？**

> ffmpeg 合成耗时长，独立进程避免阻塞 Web；边界清晰：API 接单，Worker 消费。和奇觅类产品的「生成服务」分离一样。

**Q：和奇觅比做了什么、没做什么？**

> 做了：上传、配置、异步任务、进度、下载的产品流。  
> 没做：游戏广告 AI 混剪算法、投放侧能力；用的是模板 JSON + ffmpeg 规则编排。

---

## 5. 长任务 / E2E / 质量

**Q：长任务怎么设计？**

> POST 返回 jobId，status=created；Worker 改状态机；前端轮询 GET。不在 HTTP 里跑 ffmpeg。

**Q：什么是 E2E？你做过吗？**

> 从用户操作到业务结果，跨页面/API/DB/Worker/文件。我做过 **手动 E2E**：上传→等 Worker→播放下载。MVP 阶段足够；上线后可补 Playwright。

**Q：怎么保证质量 / 验收？**

> 每层有标准：POST 后 Studio 1 Job + N Asset；Worker 后 completed + outputs 有 mp4；任务页可播放。这叫 **验收标准**，不是只看代码编译过。

---

## 6. 数据 & 存储 & Prisma

**Q：Prisma 是什么？**

> TS 侧 ORM，schema 定义表，create/find 生成 SQL 操作 Postgres；migrate 进 git 保证环境一致。

**Q：文件怎么存？**

> multipart 上传；元数据 Job/JobAsset；文件写 storage/uploads/{jobId}/；DB 存相对路径 uploads/.../photo_000.jpg。

**Q：为什么 Job 和 JobAsset 两张表？**

> 一对多：一个任务多张照片；Worker 按 sortOrder 读路径合成。

---

## 7. 排查问题怎么说

> 分层：Network → Next 终端 → Worker 终端 → Prisma Studio（status、errorMsg）→ storage 文件是否存在。  
> 和前端调试类似，没有 DOM，靠 **日志 + 数据一致性**。  
> 例：一直排队 = Worker 没跑；POST 500 = 看 Next 终端和 migrate；failed = errorMsg + Worker traceback。

| 前端习惯 | 本项目 |
|----------|--------|
| Chrome DevTools | Network + 三终端 + Studio |
| 看 state | Job 表字段 |
| 断点 | console.log / print |

---

## 8. 工程化：Redis / Kafka / OSS

**Q：Redis？**

> 第二周计划：API push jobId，Worker pop，替代 DB 轮询。容器已 docker compose 起了，Worker 还没接。

**Q：Kafka？**

> MVP 不用。海量、多服务、消息回放场景才需要；个人项目 Redis 足够。

**Q：OSS？**

> 本地 storage 的云端版；第二周上线；DB 仍存 key，下载用签名 URL。

**判断力一句：**

> MVP 选能跑通的最简方案；知道何时升级 Redis 和 OSS。

---

## 9. 全栈转型 & 和 AI 协作

**Q：前端怎么转全栈？**

> 从用户流程和接口契约出发，先画上传→轮询→下载，再补 API、Prisma、Worker。同一条业务链扩展，不是先学一门后端语言。

**Q：AI 协作不空转？**

> 要做什么 + 成功标准 + 怎么验证。例：Worker 接入后 E2E 必须到 mp4。我验收 Network/终端/Studio/磁盘四层。

## 15. E2E 端到端（讲清楚这个词）

- **端** = 用户感知的起点（上传）和终点（拿到 mp4），不是前端模块→后端模块  
- **本质** ≈ 完整业务流程；E2E 是测试领域的标准名，强调跨层 + 以用户结果为准  
- **手动 E2E** = 自己点浏览器（你已做过）  
- **金句**：E2E 验的是用户故事整段成立，不是 API 返回 200  

详见曾单独整理的 [面试话术与概念速查](./面试话术与概念速查.md) 第 1～2 节（本章已浓缩）。
