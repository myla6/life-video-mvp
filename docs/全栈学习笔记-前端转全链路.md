## 1. 一句话地图

```text
前端：发单 + 查物流（提交 + 轮询 + 预览下载）
Next API：前台登记 + 校验 + 写库 + 存文件
Postgres：账本（Job / JobAsset / status）
storage/：仓库（照片、成片 mp4）
Python Worker：工厂（轮询 DB → ffmpeg 合成 → 改 status）
Prisma：用 TypeScript 操作 Postgres，不用手写 SQL
```

**核心句（面试可用）：**

> 生活纪念短片 MVP：Next.js 负责上传与任务 API，Postgres 持久化任务状态，Python Worker 异步消费任务并用 ffmpeg 合成竖屏 mp4；文件存本地磁盘，第一周 DB 轮询，第二周演进 Redis 与 OSS。

## 2. 前端 vs 全栈：心智模型转换

### 以前（公司 / 纯前端）

```text
React 页面 → fetch 公司后端 API → 等响应 → 展示
```

你只负责调用方：知道传什么、返回什么、怎么轮询。

### 现在（Next.js 全栈 MVP）

```text
React 页面 → fetch /api/jobs（同项目里的 route.ts）→ Prisma / 磁盘 / Worker
```

**`app/api/**/route.ts` 就是后端**，相当于 Java 的 Controller，只是和前端在同一个 repo。

### 要建立的 6 条意识（比学 Java 语法重要）

| # | 意识 | 含义 |
|---|------|------|
| 1 | **请求分两种** | 短请求（创建任务、查状态）vs 长任务（合成视频，分钟级） |
| 2 | **API 四件事** | 接参数 → 校验 → 写库/存文件 → 返回（不在创建接口里等合成） |
| 3 | **数据分两类存** | 结构化信息放 DB；大文件放磁盘/OSS |
| 4 | **ORM 是中间层** | schema 定义表 → Prisma create/find → 生成 SQL → Postgres 执行 |
| 5 | **异步 = 三件套** | 提交接口 + 状态查询 + 后台消费者（Worker） |
| 6 | **全栈 = 多进程协作** | Web 进程、DB、Worker 进程、文件存储，结构上线后不变，只是换云 |

### 寄快递类比（整链路）

| 角色 | 项目里 | 干什么 |
|------|--------|--------|
| 顾客填单 | 创作页 | 选照片、填宝宝名、提交 |
| 前台收件 | POST /api/jobs | 校验、登记编号、照片进仓库 |
| 登记本 | Postgres | 记编号、状态、路径 |
| 仓库货架 | storage/uploads/ | 真实照片 |
| 工厂 | Python Worker | 领「待生产」单、ffmpeg 加工 |
| 成品区 | storage/outputs/ | 成片 mp4 |
| 查物流 | GET /api/jobs/:id | 好了没？ |
| 看进度 | 任务页 | 每 2 秒轮询 |

## 3. 点击「生成短片」之后发生了什么

### 谁负责什么（不是「点击之后全是后端」）

| 环节 | 负责 | 语言/位置 |
|------|------|-----------|
| 表单、上传、跳转、轮询 UI | 前端 | `app/create/`、`app/jobs/` |
| 创建任务、查状态、下载 | Next API | `app/api/jobs/` |
| 持久化任务与素材清单 | Postgres + Prisma | Docker 里的数据库 |
| 照片/mp4 文件 | 本地磁盘 | `storage/` |
| 视频合成 | Worker | `worker/main.py` + `render.py` |

### 逐步流程

```text
1. 前端：FormData（文字 + 照片）→ POST /api/jobs
2. API：校验 → prisma.job.create（status=created）
3. API：照片写到 storage/uploads/{jobId}/
4. API：prisma.jobAsset.createMany（每条照片一行）
5. API：立刻返回 { id, status: "created" }（不等合成）
6. 前端：跳转 /jobs/{id}，每 2s GET /api/jobs/{id}
7. Worker：轮询 DB，认领 created → preprocessing → rendering
8. Worker：读 JobAsset 的 filePath，ffmpeg 合成
9. Worker：写 storage/outputs/{jobId}.mp4，UPDATE status=completed
10. 前端：看到 completed，<video> 预览 + 下载
```

### 和「旧理解」的修正

❌ 创建接口里合成完再返回  
✅ 创建只接单；合成由 Worker 异步做；前端通过查状态拿结果

## 4. Prisma Studio 里看到的数据

### 一次提交 5 张照片 ≈ 6 行数据

```text
Job × 1          → 一张「订单」
JobAsset × 5     → 五张「物品清单」（每张照片一行）
```

### Job 表（任务本身）

| 字段 | 含义 |
|------|------|
| id | 全链路唯一编号（文件夹名、轮询、成片文件名） |
| status | 状态机：created → … → completed / failed |
| progress | 进度条数字 |
| babyName / eventDate / blessing / bgmPreset | 表单信息 |
| outputPath | 成片相对路径，如 outputs/xxx.mp4 |
| outputUrl | 前端下载地址，如 /api/jobs/xxx/download |
| errorMsg | 失败原因 |

**Job 里没有照片二进制**，只有元信息。

### JobAsset 表（素材清单）

| 字段 | 含义 |
|------|------|
| jobId | 属于哪个 Job |
| type | photo / video |
| filePath | 相对 storage 的路径，Worker 靠它找文件 |
| sortOrder | 播放顺序 0,1,2… |
| fileName | photo_000.jpg 等 |

### 三处要对得上（验收用）

```text
① Prisma Studio：Job 1 条 + JobAsset 5 条
② storage/uploads/{jobId}/：5 个 jpg 文件
③ 若已完成：storage/outputs/{jobId}.mp4 + Job.status = completed
```

## 5. Prisma 和 SQL 怎么对应

Prisma = **用 TypeScript 操作数据库的中间层**（类似 JPA / MyBatis，库不同）。

```typescript
// 你写的
await prisma.job.create({ data: { babyName: "小糯米", status: "created" } })

// 大致等价 SQL
INSERT INTO "Job" ("id", "babyName", "status", ...) VALUES (...);
```

```typescript
await prisma.job.findUnique({ where: { id } })

// 大致等价
SELECT * FROM "Job" WHERE id = $1;
```

- **表结构**：`prisma/schema.prisma`（相当于建表 DDL）
- **迁移历史**：`prisma/migrations/*.sql`（提交 git，各环境执行同一套，不要手改已跑过的 migration）
- **改表**：只改 schema.prisma → `pnpm db:migrate` 生成新 migration

**我们用的数据库：PostgreSQL**（跑在 Docker 容器 `life-video-postgres`，本机端口 **5433**）。

## 6. 数据存哪：数据库 vs 磁盘 vs Docker

```text
你的 Mac
├── Docker 容器
│     └── PostgreSQL（Job、JobAsset 表）  ← 结构化、要查询
├── 项目 storage/
│     ├── uploads/{jobId}/照片            ← 大文件
│     └── outputs/{jobId}.mp4             ← 成片
└── apps/web（Next 进程）
      worker/（Python 进程，另开终端）
```

- **不是「没容器」**：数据库就在 Docker 里；照片/mp4 在项目文件夹。
- **第二周**：`storage/` → 换成 OSS；Docker 本地 → 换成 ECS 上部署。**结构不变**。

## 7. 长任务与状态机

### 状态流转

```text
created → preprocessing → rendering → completed
                              ↘ failed
```

| status | 用户看到 | progress 约 |
|--------|----------|-------------|
| created | 排队中 | 0 |
| preprocessing | 处理素材 | 30 |
| rendering | 合成中 | 60 |
| completed | 成功 | 100 |
| failed | 失败 | — |

### 三个角色

```text
POST /api/jobs     → 生产者（创建 created）
Worker             → 消费者（改 status）
GET /api/jobs/:id  → 查询（前端轮询）
```

这和奇觅「提交生成 → 等进度 → 下载」是同一类产品逻辑。

## 9. 后端怎么排查 Bug（对照 Chrome 调试）

前端习惯：Elements、Network、断点、看页面。

后端没有「页面」，用 **分层 + 日志 + 数据核对**。

### 排查顺序（本项目固定四站）

```text
1. 浏览器 Network
   → POST/GET 状态码？Response body 里 error/message？

2. Next 终端（pnpm dev 那个窗口）
   → POST /api/jobs failed: ... 
   → 500 多半是 Prisma、写盘、校验

3. Worker 终端（python main.py）
   → [worker] 处理任务 / 失败 / traceback
   → 合成、ffmpeg、素材找不到

4. 数据核对
   → Prisma Studio：Job.status、errorMsg、JobAsset 几条
   → ls storage/uploads/{jobId}/
   → ls storage/outputs/
```

### 和前端调试的对照

| 前端 | 后端（本项目） |
|------|----------------|
| Chrome DevTools | Network + 三个终端 + Prisma Studio |
| 看 Component state | 看 Job 表字段 |
| 断点 debugger | 临时 console.log / print（MVP 够用） |
| 看接口 Response | 同 Network；或直接 curl /api/jobs/:id |
| 样式错了 | 文件路径错、status 没变、Worker 没跑 |

### 常见问题速查

| 现象 | 先查 |
|------|------|
| 一直排队中 | Worker 是否在跑？Docker 是否 up？ |
| POST 400 | Network 里 message；多半是照片数量/格式 |
| POST 500 | Next 终端；是否 db:migrate、.env |
| status=failed | Studio 的 errorMsg + Worker 终端 |
| 没 BGM | assets/bgm/ 是否空（可无声出片） |
| 连接数据库失败 | Docker、端口 5433、worker/.env |

## 10. 工程化关键词：Redis / Kafka / E2E / 日志

### 在本项目里何时接触

| 概念 | MVP 用吗 | 何时 | 一句话 |
|------|----------|------|--------|
| **E2E 端到端** | ✅ | 已在手动做 | 从上传到下载整条链路通 |
| **DB 轮询** | ✅ 现在 | Day 4 | Worker 每 3s 查 created |
| **Redis 队列** | 第二周 | 工程化 | 替代轮询，任务 push/pop |
| **OSS** | 第二周 | 工程化 | storage 换云 |
| **ECS 部署** | 第二周 | 工程化 | 线上跑 Web+Worker+DB |
| **日志** | 简单版已有 | print/console | 第二周可结构化 |
| **Kafka** | ❌ 不用 | 产品量大后 | 比 Redis 更重，微服务常用 |

### Redis vs Kafka（面试被问到时）

| | Redis 队列 | Kafka |
|--|-----------|-------|
| 复杂度 | 低 | 高 |
| 我们 MVP | 第二周可能上 | 不需要 |
| 典型场景 | 单机/小团队任务队列 | 海量、多服务、消息回放 |

### E2E 两种

- **手动 E2E**：自己点网页（你现在做的）
- **自动 E2E**：Playwright 脚本（第二周可选）

## 11. 和 AI 协作：怎么指挥才不空转

### 不要盲指挥

❌ 「继续做 Day 6」  
✅ 「加标题卡：宝宝名+满月快乐，前 5 秒；验收：mp4 能看到字；失败时 status=failed」

### 每次改完问 AI 三句话

1. **动了哪几个文件？各干什么？**（只要一句话）
2. **我怎么手动验证？**（命令 + 网页步骤）
3. **怎样算失败？**（Studio、终端、Network）

### 你要定 vs 交给 AI

| 你来定 | 可交给 AI |
|--------|-----------|
| 做不做、验收标准 | 语法、库、ffmpeg 参数 |
| 范围（别一次加支付） | boilerplate、修报错 |
| 朋友需求 | 对照开工包拆步骤 |

### 成长 = 会验收，不是会背每一行代码

**验收（Acceptance Criteria）**：什么叫「这步做完了」——可观察、可重复的结果。

例：

- 提交 5 张照片 → Studio 里 1 Job + 5 JobAsset
- Worker 跑完 → status=completed，outputs 下有 mp4
- 任务页能播放并下载

## 13. 面试叙事模板

### 30 秒版

> 我做了一个生活场景纪念短片 MVP。前端 Next.js 上传照片和表单，Route Handler 创建异步任务写 Postgres，文件落本地 storage；Python Worker 轮询任务表，用 ffmpeg 按模板 JSON 合成竖屏 mp4，前端轮询状态完成后预览下载。第一周用 DB 轮询打通全链路，数据库 Postgres 跑 Docker，ORM 用 Prisma。

### 被问「长任务怎么设计」

> 创建接口只返回 jobId，status 初始 created；独立 Worker 消费任务更新状态机；前端轮询查询接口。避免在 HTTP 请求里跑 ffmpeg。后续可把 DB 轮询换成 Redis 队列，文件从本地迁 OSS。

### 被问「文件怎么存」

> 元数据放 Postgres，文件放对象存储（MVP 阶段本地目录）。DB 存相对路径，Worker 和 API 按约定解析。上传用 multipart/form-data。

### 被问「你怎么排查问题」

> 分层：Network 看 HTTP → API 终端看服务端错误 → Worker 看合成 → Prisma Studio 对 status 和 errorMsg，再核对磁盘文件是否存在。和前端类似，只是没有 DOM，靠日志和数据一致性。

### 被问「Prisma 是什么」

> TypeScript 侧的 ORM，schema 定义表结构，代码里 create/find 生成 SQL 操作 Postgres，迁移文件进 git 保证环境一致。
