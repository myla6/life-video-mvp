## 1. E2E 端到端：到底啥意思、为啥不直接说「完整流程」

### 1.1 哪端到哪端？

**不是「前端这一端 → 后端那一端」。**

E2E 的 **端（End）** 指的是 **用户/业务能感知到的起点和终点**：

```text
起点端：用户上传照片、点「生成短片」
终点端：用户看到 mp4、能播放、能下载
```

中间经过页面、API、Postgres、Worker、ffmpeg、磁盘——**全都算在这条链里**，但 E2E 不关心你逐层怎么实现，只关心：

> **像真实用户一样走一遍，最终目标有没有达成。**

所以你说「完整流程」在 **意思上** Often 是对的；行业里仍用 E2E，是因为有更具体的边界（见下）。

---

### 1.2 为啥专门取这个名字？

**不是近年才有的新词**，几十年前软件工程里就有 **End-to-End Test**。

名字来自 **通信/系统工程**：信号从 **一端进、另一端出**，中间每一 hop 都要通，不能假设中间「应该没问题」。

搬到软件里：

```text
End = 系统边界上真实输入 / 真实输出
     （用户的操作 / 用户拿到的结果）

不是 End = 前端模块 / 后端模块
```

---

### 1.3 和「完整流程」有啥区别？为啥不直接说人话？

| 你说 | 面试官听到 | 差别 |
|------|-----------|------|
| 「我测了完整流程」 | 偏口语，范围模糊 | 可能只测了接口，也可能测了 UI |
| 「我做了手动 E2E」 | **从用户入口到业务结果** 验过 | 范围清晰：跨多层、以用户目标为准 |
| 「我写了 E2E 自动化」 | Playwright/Cypress 那条链路 | 还暗示可重复、可回归 |

**E2E 多出来的信息：**

1. **跨层**：不是单测一个 API 或一个函数  
2. **以用户/业务结果为准**：不是「代码跑完」，是「用户目标达成」  
3. **测试语境**：常和单元测试、集成测试放在一起说，便于沟通专业边界  

**面试实用结论：**

- 跟非技术朋友：说 **「我从上传到下载整条链路跑通了」** 完全 OK  
- 跟面试官：说 **「手动 E2E 跑通」** 或 **「端到端链路验证」** 更精确、显得你知道测试分层  

不矛盾：**本质就是完整业务流程；E2E 是工程里给这类验证起的标准名字。**

---

### 1.4 手动 E2E vs 自动 E2E

| | 手动 E2E | 自动 E2E |
|--|----------|----------|
| 怎么做 | 浏览器自己点 | Playwright 等脚本 |
| 你做过吗 | ✅ 上传→等→下载 mp4 | ⬜ 第二周可选 |
| 算不算 E2E | ✅ | ✅ |

**话术：**

> MVP 阶段用手动 E2E 验证主路径；上线前可把核心路径补成 Playwright 自动化回归。

---

### 1.5 什么场景可以说「端到端」？

**✅ 适合说**

- 「我负责 **从上传到成片下载** 的端到端链路。」
- 「Worker 接入前只能 E2E 到排队；接入后 **E2E 到可播放 mp4**。」
- 「Day 7 用朋友真实素材做一条 **端到端真实成片**。」

**❌ 不太准确**

- 「写了个 POST 接口，E2E 完成了。」→ 应说 **接口层** 或 **集成**，还差用户看到 mp4  
- 「接了 Prisma，端到端 OK。」→ 只是链路中的一环  

**金句（背这一句就够）：**

> **E2E 验的是用户故事整段成立，不是单个 API 返回 200。**

## 2. 概念归属：E2E 是测试还是后端？

**主要是测试/质量领域的词**，不是「后端专属技能」。

```text
         ┌─────────────────────────────────────┐
         │  E2E 测试（质量 / QA / 全栈验收）      │  ← 名字来自这里
         └─────────────────┬───────────────────┘
                           │ 要跑通整条链
         ┌─────────────────▼───────────────────┐
         │  前端 + API + DB + Worker + 存储     │  ← 后端是链上的层
         └─────────────────────────────────────┘
```

- **前端**常做：手动 E2E、Playwright E2E（测 UI + 接口）  
- **后端**常做：集成测试、接口测试；也 **参与** E2E（保证 Worker、队列通）  
- **全栈**：天然适合说 E2E——因为你本来就要串所有层  

你之前以为是后端词、又以为是测试场景—— **两种理解都沾边，它更准确是「测试类型」，全栈/后端都要为 E2E 服务。**

## 3. 关键词一句话（面试版）

| 词 | 一句话 |
|----|--------|
| **E2E 端到端** | 从用户操作到业务结果，跨所有层验整条链 |
| **验收标准** | 什么叫「这步做完了」——能观察、能重复的结果 |
| **长任务** | 创建接口立刻返回；合成异步；前端轮询状态 |
| **状态机** | 用 `status` 字段描述任务阶段（created→completed） |
| **生产者 / 消费者** | API 创建任务；Worker 消费任务 |
| **ORM / Prisma** | 用 TS 操作 DB，不用手写 SQL |
| **元数据 vs 大文件** | DB 存路径和状态；磁盘/OSS 存照片和 mp4 |
| **DB 轮询** | Worker 定时查 `status=created`（MVP 简单方案） |
| **Redis 队列** | 任务 push/pop，比轮询 DB 更标准（第二周） |
| **Route Handler** | Next.js 里的后端 API 文件 |
| **multipart** | 带文件上传的表单格式 |
| **手动 E2E** | 自己点浏览器验全链路 |
| **集成测试** | 多层连在一起测（如 API+DB），范围比 E2E 窄 |

## 4. 项目介绍话术

### 4.1  elevator 30 秒（开场）

> 我做了一个宝宝满月纪念短片的 MVP。用户上传照片、填宝宝信息、选 BGM，一键生成竖屏 mp4。  
> 前端 Next.js，API 用 Route Handler 创建异步任务写 Postgres，文件落本地 storage；Python Worker 轮询任务表，ffmpeg 按模板 JSON 合成；前端轮询状态完成后预览下载。  
> 第一周用 DB 轮询打通 **手动 E2E**，数据库 Postgres 跑 Docker，ORM 用 Prisma。

### 4.2 60 秒（追问架构）

> 这是典型的 **长任务 + 状态机** 架构：POST 只接单返回 jobId，不在 HTTP 里跑 ffmpeg。Worker 独立进程消费任务，状态从 created 到 completed。  
> 数据上分 **Job 表** 存任务元信息和状态，**JobAsset 表** 存每张照片的路径和顺序；真实文件在 storage，DB 只存相对路径。  
> 我前端背景，这次补了 API、Prisma、Docker 本地 Postgres，以及和 Python Worker 的协作边界。

### 4.3 和奇觅类比（你有游戏广告视频经验时）

> 奇觅是上传→配置→生成→等待→下载；我这个 MVP 复用的是 **产品流程和等待体验**，不是复刻算法。实现上是 **模板 + ffmpeg + 任务流**，适合个人全链路落地。

## 5. 分题型标准答法

### Q：长任务 / 视频合成怎么设计？

> 创建接口只返回 jobId，status 初始 created；独立 Worker 消费任务并更新状态机；前端轮询 GET 任务详情。避免在 HTTP 请求里跑 ffmpeg，否则超时且无法扩展。MVP 用 DB 轮询；量上来可换 Redis 队列，存储从本地迁 OSS。

### Q：文件 / 照片怎么存？

> 元数据放 Postgres，大文件放对象存储——MVP 阶段是本地 `storage/`。DB 存相对路径如 `uploads/{jobId}/photo_000.jpg`，Worker 和 API 按约定解析。上传用 multipart/form-data。

### Q：Prisma 是什么？和 SQL 关系？

> TypeScript 侧的 ORM。`schema.prisma` 定义表结构，`prisma.job.create` 这类 API 会生成 SQL 操作 Postgres。迁移文件进 git，各环境执行同一套 migrate，保证表结构一致。

### Q：为什么 Next 当后端，不用 Java？

> MVP 个人项目，Next Route Handler 和前端同 repo，改 API 和页面一次提交，学习成本低。合成重活用 Python+ffmpeg。上线后仍是 Web 接请求、Worker 干重活，只是部署到 ECS，结构不变。

### Q：Job 和 JobAsset 为什么分两张表？

> Job 是「订单」：谁、什么状态、成片在哪。JobAsset 是「物品清单」：每张照的路径和 sortOrder。一对多，Worker 按 sortOrder 读路径做轮播。

### Q：你怎么做 E2E / 怎么保证质量？

> MVP 用 **手动 E2E**：上传 5 张照片→等 Worker→status completed→能播放下载。同时用 Prisma Studio 对数据、看 storage 目录是否和 DB 一致。后续可加 Playwright 自动化核心路径。

## 6. 排查问题怎么说

### 6.1 标准答法

> 我按层排查：浏览器 Network 看 HTTP 状态码和 body；Next 终端看 API 报错；Worker 终端看 ffmpeg 和 traceback；Prisma Studio 看 status 和 errorMsg；最后核对 storage 里文件是否存在。和前端调试类似，只是没有 DOM，靠 **日志 + 数据一致性**。

### 6.2 对照表（展示你「有方法论」）

| 前端习惯 | 全栈 MVP |
|----------|----------|
| Chrome DevTools | Network + 三终端 + Studio |
| 看 state | 看 Job 表字段 |
| 断点 | console.log / print（MVP 够用） |
| 接口 Response | Network 或 curl |

## 7. 工程化名词：Redis / Kafka / OSS

### Q：为什么第一周 DB 轮询，第二周 Redis？

> MVP 优先 **少依赖、快打通**。Worker 每 3 秒查 created 任务，单进程够用。量上来或要多 Worker 时，改成 Redis 队列：API push jobId，Worker blocking pop，减少空转和 DB 压力。

### Q：Kafka 呢？

> 我们 MVP 没用 Kafka。Kafka 适合海量、多服务、要消息回放；个人项目 **Redis 队列** 足够。我知道 Kafka 是更重一档的消息方案，等规模或微服务场景再深入。

### Q：OSS 是什么角色？

> 本地 `storage/` 的云端版。照片和 mp4 放 OSS，DB 仍存 key/路径，下载用签名 URL。第二周上线时会从本地迁过去。

**一句话展示判断力：**

> MVP 选 **能跑通的最简方案**；DB 轮询、本地盘都是刻意的；知道何时升级到 Redis 和 OSS。

---

## 8. 全栈转型 / 和 AI 协作

### Q：前端怎么转全栈的？

> 从熟悉的 **用户流程和接口契约** 出发，先画清上传→任务→轮询→下载，再补 API 实现、Prisma 写库、Worker 边界。不是先学 Java，而是 **同一条业务链** 上扩展职责。用 AI 写 boilerplate，我负责 **验收标准、E2E 和数据对账**。

### Q：你怎么用 AI 还不空转？

> 我不说「做下一步」，我说 **要做什么 + 成功标准 + 怎么验证**。例如：Worker 接入后，手动 E2E 必须从上传到 mp4 下载；Studio 里 status 变 completed。AI 改完我按 Network、终端、Studio、storage 四层验收。

## 9. 诚实边界：MVP 没做什么（加分项）

主动说 **没做什么**，比吹 **全做了** 更可信：

> 当前成片是 **照片轮播 + BGM**，标题卡、结尾祝福语、视频片段拼接、转场还在 Day 6。  
> 没有用户登录和支付；没有 Kafka；没有上 OSS，文件在本地。  
> 第一周故意 **DB 轮询** 而不是 Redis，优先验证 **E2E 业务闭环**。

面试官会觉得你知道 **范围** 和 **演进路径**。

## 11. Agent 是什么：调大模型算不算 Agent 开发

### 11.1 一句话

**Agent = 大模型 + 自主决定下一步 + 调用外部工具 + 根据结果循环，直到任务完成。**

**调一次大模型 API ≠ Agent 开发**，那是 LLM 应用 / Prompt 工程。

### 11.2 四种形态（由简到繁）

| 形态 | 流程 | 算不算 Agent | 本项目对应 |
|------|------|--------------|------------|
| 单次 LLM 调用 | 问一句 → 答一句 | ❌ | 让 LLM 写祝福语 |
| LLM + 固定流程 | LLM 出 JSON → **你的代码**固定提交 Job | ⚠️ 弱 / LLM 应用 | ROADMAP Phase 5A |
| Tool-using Agent | LLM **自己选**调哪个工具、多步循环 | ✅ | Phase 5B：analyze → pick_bgm → render_video |
| Multi-Agent | 多个 Agent 分工 | ✅（过重） | MVP 不做 |

### 11.3 四个自检问题（面试前过一遍）

1. 模型会不会**自己决定**下一步？（还是你写死 if/else）
2. 会不会**调外部工具**？（DB、API、ffmpeg Worker……）
3. 会不会**根据工具结果再推理**？（循环，不是一问一答）
4. 有没有**明确目标和终止条件**？（成片成功 / 失败告知）

**只满足「调了 LLM」→ 说「LLM 增强」更准确；满足 2～3 条 → 可以说 Agent 编排。**

### 11.4 和 ffmpeg 的分工（别混为一谈）

| 谁 | 干什么 |
|----|--------|
| LLM / 视觉模型 | 理解诉求、选片、选 BGM、生成文案 |
| **ffmpeg（Worker 工具）** | 按时间轴拼片、编码、混音、烧字幕 |

Agent 越强，**工具越多**；不是用 LLM 直接生成整段 mp4、去掉 ffmpeg。详见 [技术方案 §15.5](./技术方案-MVP全栈架构.md#155-agent-阶段还要-ffmpeg-吗和大模型怎么分工)。

### 11.5 面试怎么说（按阶段）

| 阶段 | 说法 |
|------|------|
| 现在 | 「异步任务 + Worker + ffmpeg 全栈，LLM 还没接。」 |
| Phase 5A 后 | 「LLM 结构化输出，自然语言转 Job 参数，属于 LLM 增强入口。」 |
| Phase 5B 后 | 「Agent 用 LLM 编排工具链，成片仍由 Worker + ffmpeg 执行。」 |

❌ 「用了 ChatGPT API 所以是 Agent 项目。」  
✅ 「Agent 层负责理解和编排，重编码在 Worker，边界清晰。」
