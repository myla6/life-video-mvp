## 1. 你先搞清楚：问题不在「会不会」，而在「值不值得信」

很多人其实已经比平均水平往前迈了一步：**独立做完 + CI/CD + 外网发布**。

这说明你不是「不会落地」，而是现在的作品**还不足以支撑你在面试里讲出「我能负责一块业务」的自信**。

这两个问题不一样：

| 维度 | 含义 |
|------|------|
| **会不会做** | 技术能力：写代码、部署、跑通流程 |
| **值不值得信** | 业务可信度：能否证明你能独立负责一块真实业务 |

**结论：不是「项目不行」，是「项目类型和完成度」还没到面试自信线。**

## 2. 全链路交付能力，到底在培养什么？

不是「会写前端 + 会写后端 + 会部署」这么简单。

招聘方想看到的是你能走完这条链：

```
真实问题 → 需求边界 → 方案取舍 → 数据/接口设计 → 实现 → 上线 → 观测 → 迭代/修 bug
```

每一项你都要能**讲出为什么**，而不只是「我做了」。

### 2.1 自测表（每项 0–2 分）

| 能力 | 0 分（demo 感） | 1 分（做过） | 2 分（能面试讲） |
|------|----------------|-------------|-----------------|
| 需求 | 功能堆砌 | 有明确用户场景 | 能说清不做什么、为什么砍功能 |
| 数据 | 假数据/写死 | 真数据库 | 表结构能解释字段为什么这样设计 |
| 鉴权权限 | 没有或摆设 | 有登录 | 角色/权限/越权考虑过 |
| 稳定性 | 本地能跑 | 线上能访问 | 有日志、错误处理、限流等 |
| 可维护 | 一次性代码 | 有 README | 有架构说明、环境区分、迁移方案 |
| 业务闭环 | 展示型 | 有完整主流程 | 用户能完成一个真实任务 |
| 迭代 | 做完就扔 | 修过 bug | 有版本记录、踩坑总结 |
| 差异化 | 同质化 todo | 有一点特色 | 别人一眼知道你为什么做 |

**常见情况：部署 2 分，但需求/业务闭环/差异化只有 0–1 分。**

### 2.2 怎么培养？刻意练习 4 件事

不要「学完全部再做一个」，而是：

#### ① 从一个真实痛点出发，不是从技术出发

| 错误起点 | 正确起点 |
|---------|---------|
| 「我用 Next.js 做个后台」 | 「小公司运营每周要花 2 小时整理投放数据，我想把这件事压到 10 分钟」 |

#### ② 强制自己做「产品负责人」

每个功能写三句话再写代码：

- 谁用？
- 用完省什么/得到什么？
- 失败会怎样？（没数据、重复提交、权限错了…）

#### ③ 故意留 20% 难度给自己

AI 可以帮你写 CRUD，但你要自己决定：

- 表结构
- 权限模型
- 状态机（订单/审核/发布）
- 异常和边界

#### ④ 上线后必须「运营一周」

- 自己每天用，或找 3–5 个真实用户用一周
- 修至少 3 个真实问题

**这一步是从 demo 变成作品的分水岭。**

## 3. AI 时代，程序员做什么产品才合适？

**原则：不要做「证明我会编程」的产品，要做「证明我能解决一个具体场景」的产品。**

### 3.1 适合前端出身、能体现全链路的 6 类方向

| 方向 | 为什么适合前端 | 示例（要小、垂直） |
|------|---------------|-------------------|
| 运营/投放工具 | 本职相近，业务讲得通 | 多渠道素材管理、投放数据汇总、落地页 A/B 配置 |
| 小程序 + 轻后台 | 国内场景真实 | 门店预约、活动报名、签到核销 |
| AI 工作流工具 | 2025–2026 加分 | 知识库问答后台、客服话术生成、简历/JD 匹配 |
| 内部效率工具 | B 端叙事强 | 审批流、报表导出、权限管理后台 |
| 垂直行业小 SaaS | 差异化明显 | 教培课表、健身房会员、房产带看记录 |
| 开发者工具 | 程序员面试官懂 | Webhook 调试、接口 Mock 平台、表单构建器 |

### 3.2 选题 4 条硬标准（不满足就别做）

1. **你能讲清楚用户是谁**（最好就是你自己或身边人）
2. **主流程 3 步内能完成**（注册 → 核心操作 → 看到结果）
3. **有状态和数据持久化**（不是纯前端玩具）
4. **有一个「非 AI 也能成立」的核心**，AI 只是增强

### 3.3 不建议做的（面试很难有自信）

- Todo / 记账 / 天气 / 博客（除非做到极致）
- 纯聊天 AI 壳（没有业务闭环）
- 仿大厂产品（微信/Notion/飞书），永远做不完
- 只有展示页、没有后端逻辑的项目

### 3.4 结合前端背景的建议

优先做 **「你过去工作里见过、吐槽过的低效流程」** 的产品。

例如广告/营销相关方向：

- **小型投放素材库 + 审核流**
- **落地页配置 + 发布 + 数据回收**
- **运营日报自动生成（接 API + 模板）**

这类项目面试官会觉得：**你不是在玩技术，是在解决业务。**

## 4. 怎么落地，才不是「虚飘飘的 demo」？

一个能拿去面试的作品，至少满足 **「7 个实」**：

### ① 实用户（哪怕很少）

- 自己每天用，或找 3 个朋友用
- 能说出：「谁在用、用来干什么、反馈了什么」

### ② 实数据

- 真数据库，不是 localStorage 冒充
- 有初始化、迁移、备份意识

### ③ 实权限

- 至少：登录 + 普通用户/管理员
- 能演示越权场景你怎么防

### ④ 实异常

- 网络失败、重复提交、空状态、非法输入
- 不要只有 happy path

### ⑤ 实部署

- CI/CD 自动化
- 环境变量管理、生产/开发区分、域名 HTTPS

### ⑥ 实观测

- 错误日志（Sentry 或自建）
- 你知道线上出问题怎么查

### ⑦ 实叙事

README 不只是「如何启动」，而是：

- 背景问题
- 方案选择
- 架构图
- 难点与取舍
- 后续规划

**做到这 7 条，就不是 demo，是「小型生产项目」。**

## 5. 为什么现有外网项目不敢拿去面试？

很常见，通常是因为下面几条（对照自查）：

| 可能原因 | 表现 | 面试时会发生什么 |
|---------|------|----------------|
| 功能像教程 | CRUD + 登录 | 面试官：「和教程有啥区别？」 |
| 没有业务故事 | 说不清为谁做 | 你只能讲技术栈 |
| 深度不够 | 一戳就破 | 问权限、事务、并发就虚 |
| 没有使用痕迹 | 数据是空的 | 觉得你没认真做 |
| 同质化 | 看不出你的判断 | 记不住你 |
| 你自己都不用 | 缺乏信念 | 讲的时候不自信 |

**结论：问题不在 CI/CD，在于「业务可信度和技术深度叙事」。**

## 6. 两条路：救旧项目 vs 做新项目

### 路线 A：救现有项目（更快，2–3 周）

别重做，把它「产品化」：

#### 第 1 周：补业务

- [ ] 定义 1 个具体用户（比如「小团队运营」）
- [ ] 砍掉 50% 花里胡哨功能，只留主流程
- [ ] 填真实数据（realistic mock 也行）

#### 第 2 周：补深度

- [ ] 加角色权限
- [ ] 加 2–3 个边界处理
- [ ] 加操作日志或审计（很加分）

#### 第 3 周：补叙事

- [ ] 重写 README（问题 → 方案 → 架构 → 难点）
- [ ] 录 3 分钟演示视频
- [ ] 准备 5 个面试故事（见第 7 节）

### 路线 B：做一个新的「面试主打项目」（更推荐长期，4–6 周）

**技术建议（别贪多）：**

- Next.js + PostgreSQL + Prisma
- 登录：NextAuth 或自建 JWT
- 部署：阿里云/腾讯云 1 台
- CI/CD：沿用已有经验

**范围控制：**

- 只做 1 个核心场景
- 3 个页面以内（列表 / 详情 / 配置）
- 1 个 AI 增强点（可选，但要有实际任务）

## 7. 面试时怎么讲，才能从 demo 变成能力证明

不要讲：「我用了 React、Node、Docker…」

用这个结构（每个 1–2 分钟）：

### 故事 1：为什么做

> 「我之前看到 XX 场景每周浪费很多时间，现有工具太重/太贵，所以我做了一个只解决 A 问题的轻量工具。」

### 故事 2：你怎么拆需求

> 「最初想做 A/B/C，后来砍掉 C，因为用户核心是 A。边界是……」

### 故事 3：技术取舍

> 「为什么用 PostgreSQL 不用 Mongo；为什么先单体不拆微服务；为什么鉴权用 JWT/session。」

### 故事 4：线上问题

> 「上线后遇到 XX（比如重复提交/慢查询），我是这样定位和修的……」

### 故事 5：AI 怎么用

> 「AI 负责生成模板代码和测试，但我自己定了表结构、权限和接口契约。」

**能讲这 5 个，全链路交付能力就立住了。**

## 8. 具体可执行的「自信线」方案（6 周）

### 产品建议

做一个：**「小团队运营素材/落地页管理后台」**（名字随意）

#### 核心功能只要 3 个

1. 素材上传 + 标签 + 搜索
2. 落地页配置（标题/按钮/跳转）+ 预览
3. 发布记录 + 简单数据看板（哪怕先手动录入）

#### 可选 AI 增强

- 根据产品描述自动生成落地页文案
- 或根据素材自动生成投放标题候选

### 6 周节奏

| 周 | 目标 | 检查点 |
|----|------|--------|
| 第 1 周 | 定用户、定范围、画流程、建表 | 能画出主流程图和 ER 图 |
| 第 2 周 | 主流程跑通（无 AI） | 用户能完成核心任务 |
| 第 3 周 | 权限 + 异常 + 管理端 | 能演示越权防护 |
| 第 4 周 | 部署上线 + CI/CD | 生产环境可访问 |
| 第 5 周 | 自己用一周 + 修 3 个问题 | 有真实使用记录和 bug 修复 |
| 第 6 周 | README + 面试故事 + 简历描述 | 能流利讲 5 个面试故事 |

做到这里，拿出去面试的就不是 demo，而是 **「我独立交付过一块运营工具」**。

## 9. 核心问题直接回答

| 问题 | 答案 |
|------|------|
| 全链路交付怎么培养？ | 靠 **一个小而真的产品 + 自己当产品负责人 + 上线后真实使用迭代**，不是先学一堆后端知识点 |
| AI 时代做啥产品？ | 做 **你熟悉业务场景里的效率工具**，AI 做增强，不要做纯聊天壳或教程级 todo |
| 怎么落地才不是虚 demo？ | 满足 **实用户、实数据、实权限、实异常、实部署、实观测、实叙事** 这 7 条 |
| 有外网 CI/CD 项目但不敢面试？ | 说明你已经会「发布」，下一步是补 **业务叙事 + 技术深度 + 使用痕迹**。可以救旧项目，但更建议做一个更锋利的新项目当主打 |

## 10. 自测与行动清单

### 10.1 项目自信度自测

做完自测表（第 2.1 节）后：

- **总分 ≤ 8**：项目还是 demo 阶段，建议走路线 A 或 B
- **总分 9–12**：可以救，补叙事和深度
- **总分 ≥ 13**：可以当面试主打项目，重点练讲故事

### 10.2 「7 个实」检查清单

- [ ] 实用户：至少自己每天用，或 3 人用过
- [ ] 实数据：真数据库，有迁移方案
- [ ] 实权限：登录 + 角色，考虑过越权
- [ ] 实异常：网络失败、重复提交等有处理
- [ ] 实部署：CI/CD + 环境区分 + HTTPS
- [ ] 实观测：有错误日志，知道怎么查问题
- [ ] 实叙事：README 有背景、方案、架构、难点

### 10.3 面试故事准备清单

- [ ] 故事 1：为什么做（痛点 + 用户）
- [ ] 故事 2：需求怎么拆（砍了什么、为什么）
- [ ] 故事 3：技术取舍（为什么用这个方案）
- [ ] 故事 4：线上问题（怎么定位、怎么修）
- [ ] 故事 5：AI 怎么用（你负责什么、AI 负责什么）

### 10.4 简历项目描述模板

```
【项目名称】小团队运营素材/落地页管理后台

【背景】小团队运营每周花费 X 小时整理投放素材和落地页，现有工具过重/过贵。

【职责】独立负责需求分析、前后端开发、数据库设计、部署上线。

【技术栈】Next.js + PostgreSQL + Prisma + 阿里云 + GitHub Actions

【成果】
- 将素材整理时间从 X 小时降至 X 分钟
- 实现素材管理、落地页配置、发布记录完整闭环
- 支持角色权限、异常处理、错误监控
- 已上线运行 X 周，修复 X 个真实问题

【亮点】能讲清需求取舍、技术选型理由、线上问题排查过程。
```
