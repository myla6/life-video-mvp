# 和 AI 协作指南：全链路项目怎么配合

> 面向 `life-video-mvp` 这类「前端出身、要练全链路自信」的项目。  
> 核心问题不是「AI 能不能写代码」，而是 **什么时候你拍板、什么时候交给 AI、不懂的后端概念怎么学**。

## 1. 先换一个问题

不是「懂不懂 Redis」，而是 **「这个问题在解决什么」**。

做奇觅前端时，你其实已经在用「任务队列」了——只是公司后端帮你挡掉了：

```text
用户点生成 → 不能一直转圈等 3 分钟 → 要有「任务 id + 状态 + 进度」
```

**Redis / BullMQ 只是实现这个的一种办法**，不是必考题。

和 AI 配合时，**不必先成为 Redis 专家**。你要先能回答：

| 你要能说的（产品/流程层） | 例子 |
|--------------------------|------|
| 这个操作要多长时间？ | ffmpeg 合成 1～3 分钟 |
| 用户等待时要看什么？ | 排队中 → 处理中 → 完成/失败 |
| 失败了怎么办？ | 展示 errorMsg，允许重试 |
| 第一周要多稳？ | 自己 + 朋友用，不是双 11 |

AI 负责把上面翻译成「Redis 还是 DB 轮询、表怎么设计、代码怎么写」。

**你介入的时机，是「业务边界和取舍」清楚了之后**，不是「背完 Redis 文档」之后。

## 2. 三层分工（记住这个就够）

```text
第 1 层：你定「做什么、不做什么、用户怎么走」     ← 必须你拍板
第 2 层：你 + AI 一起定「方案 A/B 各有什么代价」  ← 你拍板，AI 给选项
第 3 层：实现细节（代码、命令、配置）            ←  largely 交给 AI
```

**不懂的知识点，大多在第 2 层。** 正确姿势不是「我先学完再开工」，而是：

```text
你：我的场景是 XXX，约束是 YYY，我不懂 Redis，用大白话给我 2 个方案，各说优缺点和第一周建议。
AI：方案 A … 方案 B …
你：我选 B，因为第一周我只想 XXX。写进文档，并开始实现。
```

选完以后，**用一句话写进开工包或 ROADMAP**（例如「Week1 用 DB 轮询，队列后补」）。

这就是「介入」——不是写代码，是 **把决定留下来，以后能讲 why**。

## 3. 什么时候你必须介入（6 个信号）

### 3.1 涉及「做不做」

AI 容易顺手加：登录、WebSocket、微服务、Admin 后台……

**停一下**，对照开工包 [1.4 第一版不做什么](./mvp-满月短片-开工包.md#14-第一版不做什么out-of-scope)：

- 第一版要不要登录？→ 文档说不要
- 要不要第二个模板？→ 满月跑通再说

**介入话术：**

```text
先别写代码，对照开工包 1.4，这个功能在不在范围里？
```

### 3.2 涉及「用户看到什么」

进度文案、失败提示、照片最少 5 张——这是你的主场（奇觅经验）。

**介入话术：**

```text
失败态页面要有 errorMsg + 返回创作页，不要只 toast。
```

### 3.3 AI 给了两个技术方案，你不懂

**不要装懂，也不要直接说「你看着办」。**

**介入话术（固定模板）：**

```text
我是前端，不懂 [Redis / Prisma / OSS …]。请用「用户等 3 分钟生成视频」这个场景解释：
1. 不用它，第一周怎么跑通？
2. 用了它，多解决什么问题？
3. 对我这种第一周目标，你推荐哪个？只推荐一个，并说明以后怎么升级。
```

你要的不是 Redis 教程，是 **带推荐的选择题**。

### 3.4 表结构 / 状态字段

AI 生成的表，你要能 **指着每个字段说用途**。

**介入话术：**

```text
Job 表每个字段：谁写、谁读、什么时候有值。
Job 和 JobAsset 为什么分两张表？
```

有一个字段你讲不清 → 让它改设计或改名字，**别带着糊涂进实现**。

### 3.5 第一次跑通某条链路

例如 Day 2 第一条 `final.mp4`、第一次 `POST /api/jobs` 成功。

**你要亲自跑**，看文件、看时长、看竖屏，不能只看 AI 说「完成了」。

**介入话术：**

```text
我跑完结果是 XXX，和预期 YYY 不一致，只改这一处。
```

### 3.6 AI 一次改太多文件

容易看不懂 diff，也失去「了然于心」。

**介入话术：**

```text
这次只做 X，不要动 Y 和 Z。改完告诉我改了哪几个文件、各自干什么。
```

## 4. 什么时候可以少介入

| 场景 | 你可以说 |
|------|----------|
| Prisma schema 从开工包草案生成 | 「按开工包第六节生成，字段注释写中文」 |
| ffmpeg 命令封装 | 「按模板 JSON 读 segments，输出 720×1280 mp4」 |
| 上传组件、轮询页 | 「参考开工包页面清单，样式简单即可」 |
| docker-compose、README 启动步骤 | 「对齐现有目录结构」 |
| 重复 CRUD、类型定义 | 直接给 spec 让它写 |

**原则：输入是文档里的「合同」，输出你要能验收。**

## 5. 完整协作示例：Redis 要不要上？

你不需要预先懂 Redis，可以按下面三轮对话走。

### 第 1 轮 —— 要选项，不要教程

```text
我的项目：用户上传后异步合成 60 秒视频，Worker 用 Python+ffmpeg。
我是前端，不懂 Redis。
请给第一周两个方案：
A) 只用 Postgres，Worker 轮询 status=created 的任务
B) Postgres + Redis 队列
每个方案说：我能几天跑通、缺点、什么时候必须升级。
只推荐一个，理由一句话。
```

### 第 2 轮 —— 你拍板（哪怕半懂）

```text
我选 A，第一周先 DB 轮询。
请把决定写进 ROADMAP 一行，并说明 Phase 3 何时加 Redis。
```

### 第 3 轮 —— 要「能面试讲的话」

```text
用面试口吻帮我写 3 句话：为什么第一周没用 Redis，以后怎么加。
```

这样你就有 **决策 + 文档 + 面试故事**，比「背 Redis 五大数据结构」更贴目标。

> 开工包本身也写了：Day3 可 DB 凑合，Day4 再上队列。选 A 完全合理，不是「前端不会所以将就」。

## 6. 不懂的知识点：四问模板

以后任何陌生词（Prisma、BullMQ、OSS 签名 URL、multipart）都可以问：

```text
1. 在我的项目里它解决什么问题？（一句话）
2. 不用它第一周能不能跑通？
3. 用了它我多维护什么？
4. 你推荐我现在做还是后做？为什么？
```

**你介入 = 问完这四问并做一个选择**，不是先上课再开工。

## 7. 按 Day 拆：你 vs AI（满月 MVP）

| 阶段 | 你要做的（介入） | 交给 AI 的 |
|------|------------------|------------|
| Day 1 | 访谈朋友、填开工包、能复述 In/Out Scope | 帮你整理访谈成需求 bullet |
| Day 2 | **自己跑** ffmpeg，确认 mp4 能播、竖屏、有 BGM | 写 `render.py` 或 shell 脚本 |
| Day 3 | 定：上传字段、Job 创建后 status 是什么、文件存哪 | 写 `POST /api/jobs` + Prisma |
| Day 4 | 看 Worker 是否按状态更新；失败能否在 DB 里看到 | 写 `main.py` 轮询/消费逻辑 |
| Day 5 | 自己点一遍：上传→等→预览→下载 | 轮询页、`GET /api/jobs/:id` |
| Day 6 | 定校验规则（照片 <5 张拦截） | 标题卡 ffmpeg/drawtext |
| Day 7 | **朋友真实素材**走一遍，记卡点 | 按你的卡点小步修 |

**Day 2 和 Day 7 必须你亲手做**——这是「全链路自信」的来源；AI 替不了。

## 8. AI 把你带偏的信号

- 没读开工包就建议上微服务、K8s、GraphQL
- 一次生成整个项目，你讲不清模块关系
- 表结构比开工包多很多「以后可能用到」的表
- 说「完成了」但你没本地跑过、没打开 mp4
- 用很多术语，但回答不了「用户在这一步看到什么」

**介入话术：**

```text
停。先对照 docs/mvp-满月短片-开工包.md 第一节和第六节，只做 XXX。
```

## 9. 每次开任务的会话模板（复制即用）

```text
项目：life-video-mvp，先做满月 60 秒竖屏模板。
请先读 docs/mvp-满月短片-开工包.md 相关章节。
我是前端，后端/运维概念请用业务语言解释。
本次只做：[具体一项，例如 Day2 ffmpeg 本地验证]。
不要做：[例如 Redis、登录、第二个模板]。
做完请告诉我：改了哪些文件、我怎么验收、用 3 句话说明为什么这样设计。
```

**「请先读 docs/…」+「本次只做」+「怎么验收」** —— 这三句就是介入框架。

## 10. 两个常见困惑的直接回答

### 「不知道什么时候介入」

凡是 **范围、用户可见行为、方案二选一、第一次跑通、一次改太多** —— 介入。

纯实现且验收标准在文档里写死的 —— 少介入。

### 「不懂 Redis，AI 问我怎么办」

不要答「要/不要」。让 AI **用你的产品场景给两个方案 + 一个推荐**；你选推荐或选「先简单」，并 **写进 ROADMAP 一行**。

你要练的是 **在信息不完整时敢做决定并留下理由**，不是先成为后端。
